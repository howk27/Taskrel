# Taskrel Onboarding And Form Readiness Workflow

## Goal

Complete Taskrel's onboarding workflow and simplify the core forms contractors use to set up the business, create quotes, and maintain quote document settings. The product should make every important action visibly resolved: complete, optional, needs attention, or blocked by an error. Contractors should understand what data affects client-facing documents, what stays internal, and what still needs work before they can send a quote with confidence.

## Product Context

Taskrel is a phone-first operational tool for solo and small trade contractors. Users are often entering information between job-site tasks, so forms need to be compact, direct, and stateful. The app already stores business profile data, quote document settings, logo uploads, overhead pricing inputs, and quote readiness data, but these are spread across onboarding, settings, and quote creation without one shared status language.

The current onboarding page only captures business type, trades, and primary trade. Settings contains quote document and overhead forms, but business information is mostly read-only. The new quote form has a review readiness panel after generation, but the initial quote creation step does not show completion or edge states before generation.

## Recommended Approach

Use an onboarding-first workspace setup workflow, then reuse the same form readiness model in settings and quote creation.

Onboarding should collect the business information that directly affects quoting: identity, trade profile, internal pricing defaults, and quote document defaults. Settings should become the maintenance surface for those same inputs after onboarding. Quote creation should show readiness before and after generation, so the primary action always matches the missing or completed state.

## Scope

In scope:

- Onboarding page and completion action.
- Business information editing in settings.
- Quote document settings form, including hiding the visible logo URL field.
- Internal overhead settings in onboarding and settings.
- New quote creation form and generated quote review readiness.
- Shared readiness/status helpers or small presentational components when they reduce duplicated logic.
- Copy that explains which fields affect client-facing quote documents and which stay internal.

Out of scope:

- New billing plan design.
- A full CRM redesign.
- New quote statuses in the database.
- Changing how quote line item prices are generated by AI.
- Automated migration of all historical quote business snapshots unless the user explicitly chooses to refresh a quote document before resending.

## Shared Status Model

Taskrel should use the same completion language across onboarding, settings, quote creation, quote documents, billing, and exports.

- Complete: the action or section is ready and has the data needed for the related workflow.
- Needs attention: required or high-value data is missing, invalid, or preventing the next action.
- Optional: the section can improve the workflow but should not block the user.
- Error: the app tried to save, sync, upload, send, or read required data and hit a provider, validation, or database problem.

Status labels must include text, not color alone. They should appear near the section title, in summary rows, and in the right-side or sticky readiness panel where appropriate.

## Onboarding Design

Onboarding becomes a workspace setup flow with four compact sections.

### Business Information

Required:

- Business name.
- Business type.
- Trades offered.
- Primary trade.

Optional but recommended:

- Business phone.
- Website.
- License or insured text.

The form should explain that this information appears on quote documents where relevant. Business name, phone, website, and license text should also be editable later in settings.

Completion rules:

- Complete when business name, business type, at least one trade, and primary trade are present.
- Needs attention when any required field is missing.
- Optional for phone, website, and license text.
- Error when saving fails.

### Internal Cost Defaults

Internal cost defaults belong under Business Information because they affect how Taskrel recommends quote pricing.

Fields:

- Add overhead to pricing toggle.
- Overhead percent.
- Fixed overhead per job.
- Internal preview using a sample quote, such as "On a $2,500 quote, Taskrel considers $X overhead."

Copy should avoid accounting jargon: "Use this for insurance, fuel, admin time, tools, or other costs you want Taskrel to consider when pricing. Clients will not see this as a line item."

Completion rules:

- Complete when the toggle is on and at least one overhead value is greater than zero.
- Optional when the toggle is off and both values are zero.
- Needs attention when percent is over 100, any value is negative, or a value is not numeric.
- Error when overhead columns are missing or saving fails.

### Quote Documents

Fields:

- Logo upload with preview.
- Quote template preset.
- Default client note.
- Default terms.
- Policies and warranty text.

The visible Logo URL input should be removed. The form may keep a hidden `logo_url` input so an uploaded logo remains part of the existing save flow, but users should interact with the logo only through upload, preview, replace, and remove actions.

Completion rules:

- Complete when a template is selected and at least one client-facing default exists: logo, note, terms, policy text, phone, website, or license text.
- Optional when template exists but no extra defaults are configured.
- Needs attention when an upload is in progress or an invalid image is selected.
- Error when upload or save fails.

### Start Work

The final onboarding step should not be a generic "Continue" moment. It should route the contractor to the most useful next action.

- If no quotes exist, send them to create the first quote.
- If drafts exist, send them to the quote work queue.
- If active work exists, send them to the dashboard.

The final screen should summarize setup states and show clear next actions for incomplete optional items without blocking useful work.

## Settings Design

Settings should become a compact workspace maintenance surface rather than a list of disconnected cards.

Primary sections:

- Business information.
- Quote documents.
- Internal pricing.
- Billing and payments.
- Exports and sync.

Business information should be editable in settings with the same fields used in onboarding: business name, phone, website, license text, trades, primary trade, and business type. These fields already exist on the contractor profile, so the implementation should update the existing contractor row instead of introducing a second profile model. The current read-only business rows should become a compact summary header above the editable form or be replaced by the form itself.

Quote document settings should show the logo upload and document defaults in a tighter layout. On desktop, the form should sit beside a compact document readiness summary or quote preview. On mobile, sections should collapse into short groups with status chips, but all controls must remain easy to tap.

Completion states should appear beside each settings section:

- Business information: complete when required identity and trade data exist.
- Quote documents: complete or optional using the same rules as onboarding.
- Internal pricing: complete or optional using overhead rules.
- Billing and payments: complete when subscription and Stripe Connect are active, needs attention when payment setup is missing, error for provider problems.
- Exports and sync: optional when disconnected, complete when connected, error when sync fails.

## New Quote Form Design

The new quote form should guide users before generation and after generation.

### Before Generation

The form should be organized by workflow state rather than only by input type:

- Client: name, email, phone.
- Site: address, optional property context.
- Scope: job description and additional details.
- Readiness: compact summary of what is ready and what is missing.

Completion rules:

- Client is complete when a client name and at least one send channel, email or phone, exist.
- Client needs attention when the name is missing.
- Send channel needs attention when neither email nor phone exists, but this should not block saving a draft.
- Address is optional, with copy that it improves pricing intelligence and records.
- Scope is complete when the job description has at least 20 non-space characters.
- Scope needs attention when empty or too short.

The primary action should adapt to readiness:

- Add client contact.
- Describe job.
- Generate quote.

On desktop, show a compact right-side readiness panel. On mobile, use a sticky bottom readiness/action bar so users can see what is missing without scrolling back to the top.

### After Generation

The generated quote review step should keep the existing readiness checklist pattern and make it more consistent with the shared status model.

Readiness items:

- Client contact.
- Line items.
- Totals.
- Job context.
- Send channel.
- Quote document settings.

The primary action should adapt:

- Save as draft when send channel is missing or the user chooses not to send.
- Send quote when a channel is present and required quote data is complete.
- Add contact before send when contact is missing.

Errors should distinguish saved and unsent states. If the quote saves but sending fails, the message should say the quote was saved and explain the failed channel.

## Logo And Quote Document Snapshot Behavior

Saved logo and quote document settings should clearly affect new quote previews and future sends. When a quote is sent, Taskrel may store a business snapshot on the quote. Existing sent quotes can therefore have older branding.

The product should make that behavior explicit:

- New and unsent quote documents should use current contractor settings.
- Sent quotes with a stored business snapshot should show "Using saved business snapshot."
- Quote detail should offer a refresh action before resending if the current business settings differ from the stored snapshot.
- Refreshing should update the quote business snapshot and template preset for that quote before resend.

This prevents silent mismatch between settings and client-facing quote documents while preserving historical records.

Use existing contractor fields where possible:

- `business_name`
- `business_type`
- `trade`
- `primary_trade`
- `trades`
- `email`
- `business_phone`
- `business_website`
- `license_text`
- `logo_url`
- `quote_default_terms`
- `quote_default_note`
- `quote_policy_text`
- `quote_template_preset`
- `overhead_percent`
- `overhead_fixed_per_job`
- `onboarding_complete`

Quote readiness should continue using existing quote fields:

- `client_name`
- `client_email`
- `client_phone`
- `client_address`
- `line_items`
- `subtotal`
- `tax_amount`
- `total`

- `business_snapshot`
- `template_preset`

No new database fields are required for the initial implementation. Partial onboarding progress can stay local to the client until the contractor submits the setup flow.

## Error And Edge States

Logo upload:

- Invalid file type: show Needs attention with accepted types.
- Oversized file: show Needs attention with the size limit.
- Upload failure: show Error and preserve the previous logo.
- Successful upload: show Complete and update preview immediately.

Business information:

- Missing required onboarding fields should block onboarding completion.
- Optional fields should not block progress.
- Save failures should show Error near the form and preserve entered values.

Overhead:

- Invalid values should show Needs attention before save where possible.
- Missing database columns should show Error with migration guidance.
- Zero overhead should be Optional, not incomplete.

Quote creation:

- Missing client contact should block send, not draft save.
- Missing or short job description should block generation.
- Send provider failures should distinguish email, SMS, and configuration errors.
- Saved-but-not-sent should be communicated clearly.

Historical quote documents:

- A quote using an older business snapshot should not silently change.
- A quote should be refreshable before resend when current branding differs.

## Layout And Space Use

Taskrel should maximize usable screen space by reducing stacked card height and repeated explanatory text.

Guidelines:

- Use section headers with status chips instead of large intro cards.
- Keep helper copy short and close to the field it explains.
- Use two-column field grids on desktop and one-column stacks on mobile.
- Keep readiness panels compact, scannable, and sticky when useful.
- Avoid nesting cards inside cards.
- Avoid large decorative panels in operational workflows.
- Prefer dense rows, inline previews, and small summary strips over tall standalone cards.

## Accessibility

- All statuses must have readable text labels.
- Color cannot be the only status signal.
- Touch targets should be at least 44px.
- Inputs need visible labels, not placeholders alone.
- Error messages should identify the field or section that needs attention.
- Mobile layouts should be verified at 390px.

## Testing And Verification

Automated checks:

- Run `pnpm lint`.
- Run quote workflow model tests if readiness helpers are touched.
- Run pricing tests if overhead logic is touched.
- Run quote document tests if logo or document rendering behavior changes.
- Run `pnpm build` after route-level or shared component changes.

Manual checks:

- New user completes onboarding with required business/trade fields only.
- New user completes onboarding with overhead enabled.
- New user skips overhead and sees Optional, not an error.
- Logo upload hides the URL field, updates preview, and persists.
- Quote document preview or send reflects saved logo for new/unsent quotes.
- Existing sent quote with old snapshot indicates that it is using a saved snapshot.
- Settings business information can be edited after onboarding.
- Settings sections show complete/optional/needs-attention/error states.
- New quote form shows missing client contact before send.
- New quote form allows saving draft without email or phone.
- New quote form blocks generation when scope is empty or too short.
- Mobile quote creation uses available space without hiding the primary action.

## Implementation Notes

Start with a small shared readiness model for setup sections. Keep it pure and testable, similar to the existing quote workflow model. Then update onboarding to use that model. After onboarding is complete, update settings to maintain the same fields and states. Finally, align the new quote form with the same readiness language before and after generation.

Avoid broad visual rewrites outside these forms. The goal is clearer workflow state, more compact layouts, and fewer surprises when saved settings affect quote documents.
